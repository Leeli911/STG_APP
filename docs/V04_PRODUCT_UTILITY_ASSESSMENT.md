# v0.4 产品有用性评估

更新日期：2026-07-23
状态：机制可测试，真实用户价值尚未验证

## 用户要完成的任务

目标用户不是来“得到一篇更漂亮的 AI 改稿”，而是在临时汇报、请求决策和说明理由时，快速练会三个可复用动作：

1. 先知道这次沟通要推动什么决定。
2. 第一时间给出判断，不让听众等到最后。
3. 把多个理由整理成两到三点，而不是连续堆信息。

最小使用场景是五分钟、中文、文本短答，不要求注册，不要求用户会写提示词。

## 为什么它可能比直接问 ChatGPT 更有用

| 普通对话式 AI 常见路径 | STG v0.4 的差异 |
| --- | --- |
| 用户把原稿交给 AI，得到更好的成品 | 用户先冷答，系统保留真实起点 |
| AI 一次给很多建议 | 每次只指出一个可观察结构动作 |
| 用户复制结果，所有权属于 AI | 不提供一键采用，必须亲自重写 |
| 同一稿件前后看起来更顺 | 使用未见题检查能否独立迁移 |
| 评分由黑盒模型给出 | 场景目标、规则版本和原文证据可复核 |
| 每次都要重新解释教练要求 | 固定课程与进度降低提示词和管理成本 |
| 可能产生模型费用和隐私顾虑 | 公开训练零外部模型调用，答案保留在浏览器 |

最值得突出的卖点不是“AI 更聪明”，而是：

> STG 把一次性代写，改造成有冷答、有证据、有主动重写、有未见题迁移和延迟冷测的刻意练习。

## 当前支持该假设的证据

| 证据层 | 当前结果 | 可以支持什么 |
| --- | --- | --- |
| 功能验收 | 提交、反馈、重写、迁移、刷新恢复、延迟冷测均有自动化覆盖 | 产品流程可运行，不再是静态摆设 |
| 规则评测 | 180 条冻结样本 Macro-F1 1.000；严重假通过 0/45 | 当前版本对已知输入的判断可复现 |
| 配对合成审计 | STG 组未见题迁移 55.6%，另两组均 11.1% | 训练机制值得继续验证 |
| 真实用户证据 | 暂无 | 不能声明需求、留存或学习效果成立 |

## 已确认优点

- 训练目标小，用户一次只处理一个认知动作。
- 冷答与自检避免先看答案后产生“我本来就会”的错觉。
- 反馈引用原文，并区分任务完成、规则状态、自检对齐和迁移结果。
- 主动重写阻止只改标点、复制 AI 或不改善目标结构。
- 18 个原创情境覆盖近迁移、远迁移和 24 小时延迟冷测。
- 刷新可恢复未完成答案；完成态不会阻塞次日冷测。
- 中文界面与中文训练内容一致。
- 免费公开版不依赖 OpenAI、Supabase 或第三方分析。

## 已确认缺点与风险

| 缺点 | 对用户的风险 | 后续处理 |
| --- | --- | --- |
| 规则词汇仍是有限集合 | 合理的新说法可能被低估 | 新建独立盲测集；真实错误优先扩充同义表达 |
| 只训练三个微技能 | 不能覆盖逻辑完整性、说服力、事实质量和自然度 | 在真实 Pilot 证明切口有价值后再扩课 |
| 延迟进度只保存在当前浏览器 | 换设备或清理存储后丢失 | 真实用户阶段再评估匿名账户同步 |
| 24 小时冷测已实现但无人类纵向数据 | “初步稳定”仍只是产品内规则状态 | 先做 14 天开发者自测，再做真实 Pilot |
| 合成角色知道实验意图 | 可能高估完成率和迁移 | 不将合成结果用于效果营销 |
| 规则可以判断结构动作，不能判断真实工作影响 | 用户可能把达标理解成完整能力 | 界面持续保留范围说明，报告不输出总能力分 |
| 当前场景由产品预设 | 与用户真实岗位情境可能不匹配 | Pilot 记录“找不到适合题目”的频率，再决定场景扩展 |

## 无真实 Pilot 时的验收方案

接下来可以执行两层低成本验证：

### A. 14 天开发者体验审计

- 每天只做一条当天未见题，开始前不查看规则。
- 记录完成时长、是否理解题目、卡住位置、重写次数、是否想放弃。
- 次日先完成到期冷测，再看历史结果。
- 测试期间只修阻断 Bug，不为了让分数好看而改标签。
- 第 14 天输出摩擦清单；只能称开发者体验证据。

### B. 新一轮盲测与可用性走查

- 由新的评审角色只看题目、回答与 Rubric，标注规则输出是否合理。
- 重点检查当前薄弱处：隐含请求、自然口语、短分点、无标点和表达自然度。
- 使用桌面端与移动端完成键盘、焦点、错误提示和低耐心退出路径。

## 进入真实 Pilot 前的产品门

- 首次训练从打开页面到完成迁移不超过五分钟的设计目标，经开发者自测可重复达到。
- 用户每次点击都有状态变化、明确错误或重试路径。
- 不展示无来源的“能力提升百分比”。
- 延迟冷测不会被旧完成会话阻塞。
- 新盲测无严重假通过；证据 100% 来自原文。
- Pilot 招募后首先验证：目标任务是否真实、用户是否愿意亲自重写、未见题是否被理解、次日是否愿意回来。
