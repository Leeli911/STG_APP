# v0.4 可信评测与抗投机计划

更新日期：2026-07-23
目标版本：`v0.4.0-beta.1`
状态：本轮实现与本地验收完成，待提交发布；真实用户价值仍待验证

## 1. 为什么进入 v0.4

v0.3 已经证明一个完全免费、无需登录的结构化表达训练闭环可以被实现并稳定运行，但尚未证明其判断足够可信，也尚未证明真实用户能够因此形成可保持的表达习惯。

在无法立即招募真实用户进行 Pilot 的阶段，v0.4 不急于增加更多页面、课程或 AI 能力，优先解决两件更基础的事：

1. 系统是否真的识别了目标结构动作，而不是被关键词、复制粘贴或极小修改骗过。
2. 即使没有真实 Pilot，团队是否能用可重复、可审计的合成用户实验尽早发现规则、内容和交互问题。

v0.4 的一句话目标：

> 把“流程可以完成”升级为“判断有证据、失败可解释、投机难通过、结论不过度外推”。

### 实施快照

截至 2026-07-23：

- 已建立 18 个原创场景题，覆盖冷答、近迁移、远迁移和延迟冷测。
- 已实现四状态规则、场景目标锚点、原文位置证据、有效重写和抗元评分话术。
- 已冻结 180 条四状态回归样本；当前 Macro-F1 1.000、严重假通过 0/45、证据溯源 100%、首选证据命中 97.8%。
- 已冻结 6 个行为角色、3 种训练机制、54 条严格配对轨迹；合成审计完整性门通过。
- 已实现当前会话刷新恢复、24 小时到期冷测，以及“未练习／练习中／待冷测／初步稳定”进度。
- 桌面端和移动端 Playwright 已覆盖提交、反投机、刷新恢复、不同分组和延迟冷测。
- 详细结果见 [规则评测](evals/V04_RULE_EVAL.md)、[合成用户审计](evals/V04_SYNTHETIC_USER_AUDIT.md) 与 [产品有用性评估](V04_PRODUCT_UTILITY_ASSESSMENT.md)。

这些是工程和合成输入证据，不改变“尚无真实用户 Pilot”的事实边界。

## 2. v0.3 基线

### 2.1 已确认优点

- 训练目标窄而明确：只训练明确目的、结论先行、两到三点框架三个微技能。
- 训练顺序强调主动学习：无提示冷答 → 自检 → 单点证据反馈 → 手动重写 → 未见题迁移。
- 不提供一键采用 AI 改稿，避免把“拿到好答案”误当成“学会表达”。
- 反馈引用用户原文，规则运行结果可重复、可回归。
- 中文受众和中文界面保持一致，不把中英文内容混在同一训练流程。
- 公开训练在浏览器本地运行，无需登录、Supabase 或外部模型，成本为零且答案不离开设备。
- 已建立自动化测试和桌面端、移动端端到端验收；线上提交能够进入后续训练步骤。
- 产品边界表达诚实：规则达标不等于完整沟通能力提升，自动化验收不等于用户价值验证。

### 2.2 已确认缺陷

| 缺陷 | 当前风险 | v0.4 处理原则 |
| --- | --- | --- |
| 用户填写的“核心结论”参与定义评分基准 | 用户复制任意原句也可能自我证明达标 | 场景目标和必要信息成为外部基准；自检只测认知对齐 |
| 分组规则偏重“第一、第二、首先、其次”等形式词 | 重复、无关或空洞分点可能被判为合格 | 同时检查数量、相关性、独立性和重复 |
| “发生文本变化”近似“完成重写” | 加标点或无关词也可能通过 | 只有目标行为改善且信息未被破坏才算有效重写 |
| 完成流程与技能达标混为一谈 | 完成次数容易被误读为能力证据 | 拆分任务完成、规则状态、自检对齐和迁移表现 |
| 每个技能题目数量少且迁移题固定 | 重复使用后形成记题，而非迁移 | 增加不同场景家族并避免短期重复 |
| 60 个样本与规则共同开发 | 容易对开发样本过拟合 | 建立冻结的独立 Holdout，不允许用其逐例调参 |
| 二元判断表达过度确定 | 边界输入会被迫判为成功或失败 | 新增 `uncertain`，不确定时不乐观宣称达标 |
| 只有即时迁移 | 无法讨论保持或长期学习 | 增加 24–72 小时延迟冷测机制，但不把合成结果当成人类记忆 |

## 3. v0.4 目标与非目标

### 3.1 目标

1. 建立场景约束驱动、可解释且不依赖用户自报答案的判断契约。
2. 区分 `taskCompleted`、`skillStatus`、`selfCheckAlignment` 和迁移结果。
3. 建立包含正常、边界、对抗和语言变体的冻结评测集。
4. 建立完全免费、可在 CI 中确定性执行的结构化表达 Eval。
5. 用多个职责隔离的子智能体执行样本生成、红队攻击、盲评和分歧复核。
6. 扩展原创场景，支持近迁移、远迁移与延迟冷测。
7. 每次评测输出失败案例、规则版本和可复核报告，不只输出单一准确率。
8. 保留真实 Pilot 接口；未来有条件时，可以用相同指标对照真实使用结果。

### 3.2 非目标

- 不把子智能体模拟包装成真实用户 Pilot。
- 不宣称验证了真实需求、留存、付费意愿或长期学习效果。
- 不接入付费 OpenAI 或其他按量计费模型。
- 不让外部或本地语言模型成为权威评分器。
- 不在本轮加入语音、简历/JD 解析、支付、团队协作或 AI 动态出题。
- 不以增加课程数量掩盖规则可信度问题。
- 不复制《结构化表达：如何汇报工作、演讲与写作》的原文、案例或受版权保护内容；只依据公开主题设计原创训练。

## 4. 免费与零外部模型原则

公开 App 和默认 CI 必须满足：

- 训练、评分、历史和报告均可在本地或浏览器内完成。
- 不调用 OpenAI、Supabase、第三方分析或任何收费 API。
- 不上传用户完整答案。
- 相同输入、场景版本和规则版本必须得到相同结果。
- 冻结 Fixture 是 CI 的权威输入；模型输出不能在测试运行时动态改变它。
- 子智能体可以在开发阶段帮助构造候选样本、寻找漏洞和提出评审意见，但候选内容必须经过审查后冻结进仓库。
- 如果以后加入免费本地模型，只能用于生成语言变体或建议人工复核，不能覆盖确定性规则结果。

## 5. 新的测量契约

建议核心结果至少包含：

```ts
type SkillStatus = "met" | "partially_met" | "not_met" | "uncertain";

type StructuredPracticeResult = {
  taskCompleted: boolean;
  skillStatus: SkillStatus;
  selfCheckAlignment: "aligned" | "partially_aligned" | "misaligned";
  evidence: {
    quote: string;
    start: number;
    end: number;
  }[];
  targetBehaviorImproved?: boolean;
  informationPreserved?: boolean;
  scenarioVersion: string;
  ruleVersion: string;
};
```

语义约束：

- `taskCompleted` 只表示用户完成了要求的交互步骤，不表示技能达标。
- `skillStatus` 只针对当前场景和当前微技能，不代表完整表达能力。
- `selfCheckAlignment` 衡量用户对自己表达重点的识别是否与文本和场景目标一致，不作为评分真值。
- `evidence.quote` 必须能在原文中按位置精确恢复。
- `targetBehaviorImproved` 只有当目标状态相对冷回答改善时才为真。
- `informationPreserved` 用于阻止为了通过形式规则而删除关键事实、请求或约束。
- 所有结果必须包含场景和规则版本，避免升级后无法复现。

## 6. 分阶段开发

### M1：修复判断基准 — 已完成

- 为每道题增加场景目标、听众、必要行动、必要事实和允许的表达变体。
- 用户自检结果退出权威评分路径。
- 增加 `partially_met` 与 `uncertain` 状态。
- 将训练完成与技能状态分开持久化。
- 将证据从单纯字符串升级为原文位置范围。
- 重写验收同时检查目标动作变化与关键信息保留。

退出标准：

- 复制任意原句到“核心结论”不能改变原回答的技能判断。
- 只改标点、空格或无关词不能被记为有效改善。
- 每个反馈证据均能从原文位置精确恢复。
- 所有边界输入有明确状态，不以异常或乐观默认值结束。

### M2：建立 `rule-gold-v1` — 已完成工程回归集

建立至少 180 个冻结样本：

- 三个技能各 60 例。
- 每个技能覆盖 `met`、`partially_met`、`not_met` 和 `uncertain`。
- 覆盖口语、无标点、错别字、缩写、少量英文术语和多种中文表达顺序。
- 覆盖复制题干、关键词堆砌、重复分点、无关分点、虚假自检、最小改动和删除关键信息等红队输入。
- 按场景家族划分 Development 与 Holdout；相同场景的改写不得跨集合泄漏。
- Holdout 在发布判定前冻结，不允许根据单个失败案例修改标签或规则。

标注协议：

1. 第一名盲评角色只查看场景、答案和 Rubric，独立给出状态及证据。
2. 第二名盲评角色在不知道第一份标注的情况下独立判断。
3. 第三名复核角色只处理两者分歧，并记录裁决理由。
4. 规则开发角色不能参与 Holdout 最终裁决。
5. 每次标签变更必须留下原因、日期和版本。

### M3：合成用户审计 — 已完成首轮配对审计

合成学习者角色按行为模式定义，不虚构这些角色能够代表某类真实人口：

- 背景铺垫型：先讲大量背景，结论迟到。
- 请求模糊型：描述问题但不说明希望对方做什么。
- 信息堆叠型：多个理由并列但没有分组。
- 机械模板型：存在形式标记，但分点重复或没有实际内容。
- 过度简短型：有结论但缺必要事实或依据。
- 口语无标点型：表达可理解但缺标准书面格式。
- 最小改动型：试图通过标点、同义替换或无关补字完成重写。
- 关键词投机型：堆砌“结论、第一、第二、建议”等词以骗过规则。

比较三种训练条件：

- A 组：完整 STG——证据反馈、手动重写、未见题迁移。
- B 组：直接获得一份 AI 风格改稿，不强制自己重写。
- C 组：不提供反馈，只重复作答。

采用配对设计：三组从相同冷回答出发，迁移题按场景家族和难度匹配；最终盲评角色不知道样本属于哪一组。

合成审计可报告：

- 规则的误判、漏判、边界不稳定和投机通过率。
- 反馈证据是否准确来自原文。
- 目标行为从冷答到重写是否发生可观察变化。
- 即时未见题中的目标动作是否出现。
- 关键事实是否在重写中保留。
- 不同训练机制在冻结任务上的合成表现差异。

合成审计不能证明：

- 真实用户有此需求、愿意使用或愿意付费。
- 人类能理解界面、愿意完成手动重写或会在工作中使用。
- 人类在 24–72 小时后仍然记得。
- 产品提高了真实沟通表现、留存率或职业结果。
- 某个合成角色代表真实年龄、职业、能力或文化群体。
- A 组优于 B/C 组的结果具有现实世界因果效应。

所有报告标题必须使用“合成用户审计”或“Synthetic User Audit”，不得使用“用户研究”“Pilot 完成”或“学习效果已验证”。

### M4：迁移、延迟冷测与进度 — 已完成

- 扩展为至少 18 个原创情境：3 个技能 × 6 个场景。
- 每个技能至少覆盖冷题、近迁移、远迁移和延迟冷测。
- 同一次或短期连续训练不重复相同场景家族。
- 保存当前训练步骤，刷新后能够继续。
- 进度状态采用“未练习／练习中／待冷测／初步稳定”。
- “初步稳定”至少需要间隔 24 小时的未见题达标；该状态仍只代表产品内观察。
- 使用可注入时钟自动验收时间逻辑，避免测试真实等待。

### M5：免费 Eval 工程与报告 — 已完成本地命令与报告

- 新增统一命令 `npm run eval:structured`。
- 默认只读取冻结 Fixture，完全离线运行。
- 产出机器可读 `summary.json`、失败案例清单和 Markdown 报告。
- 报告必须包含样本集版本、规则版本、Git 提交、分集合指标和红队结果。
- CI 对 Development 与 Holdout 分开统计，不用总体准确率掩盖严重失败。
- 失败报告保留原始输入、期望结果、实际结果和错误类型，便于复现。
- 付费 Smoke Test 不属于 v0.4 发布门槛。

## 7. 发布验收门槛

| 指标 | `v0.4.0-beta.1` 门槛 |
| --- | ---: |
| Holdout Macro-F1 | ≥ 0.90 |
| 严重错误被判为 `met` | ≤ 2% |
| 已知红队攻击错误通过 | 0 |
| 引用证据确实来自原文 | 100% |
| 正确证据片段命中 | ≥ 95% |
| 格式等价改写判断稳定性 | ≥ 95% |
| 相同输入、场景和规则版本的确定性 | 100% |
| 只改标点/空格被记为有效改善 | 0 |
| 公开训练的外部模型与 App API 请求 | 0 |
| 现有 Unit、E2E、Lint、Typecheck、Build | 全部通过 |

补充发布规则：

- 若 Macro-F1 达标但出现任何严重红队错误，仍不得发布。
- 若 Holdout 失败，必须先按错误类型提出通用修复，再重新冻结新版本；不得逐例硬编码答案。
- 合成审计结果必须与真实用户价值结论分栏展示。
- 发布说明必须继续保留“尚未经过真实用户 Pilot”的限制。

## 8. 没有真实 Pilot 时的替代验证

合成审计之外，保留一个不代表目标市场的单人自测方案：

- 由产品开发者连续 14 天每天完成一道未见题。
- 题目、标签和判断标准在开始前冻结。
- 记录完成时间、卡点、是否回看规则和是否放弃。
- 24–72 小时后做延迟冷测。
- 记录真实工作表达中是否主动使用目标结构动作。
- 测试期间只修复阻断性 Bug，不根据个人表现调整评分规则。

该方案能观察真实人的输入成本、遗忘和使用负担，但只能作为开发者体验证据，不能替代目标用户 Pilot。

## 9. 后续优化队列

### P0：v0.4 发布前

- 场景目标驱动评分，移除自我证明循环。
- 有效重写与信息保留判断。
- `uncertain` 和任务完成/技能达标拆分。
- 冻结 Holdout 与红队集。
- 免费离线 Eval 命令和报告。
- 至少 18 个原创场景与短期去重。

### P1：v0.4 发布后、真实 Pilot 前

- 24–72 小时延迟冷测和恢复进度。
- 失败类型可视化，解释“为什么尚未达标”。
- 键盘、焦点、移动端和低耐心路径专项审计。
- 将合成审计失败自动生成可复现测试。
- 建立规则版本迁移策略，旧历史不被新规则静默重算。

### P2：有真实用户条件后

- 5–20 名目标用户的任务观察和结构化访谈。
- 校准合成角色分布与真实错误分布之间的偏差。
- 对比完成率、即时迁移和延迟冷测，不以喜欢度代替行为指标。
- 验证用户会选择 STG、直接使用 ChatGPT 还是不训练，并记录原因。
- 只有真实使用证据支持后，才讨论课程扩展、提醒或个性化。

### Later

- 经明确同意，使用匿名真实错误扩展场景与规则。
- 免费本地模型作为语言变体生成器的可选实验。
- 文本训练被真实验证后，再评估语音表达。
- 形成真实纵向数据后，再讨论自适应复练和掌握度模型。

## 10. 事实陈述边界

v0.4 完成后可以陈述：

- 建立了版本化、可复现的结构化表达规则评测体系。
- 通过冻结 Holdout、红队样本和多角色盲评审计规则。
- 公开训练保持零外部模型调用并提供可追溯证据。
- 测量了合成输入上的判断准确性、抗投机能力和即时迁移表现。

在真实 Pilot 前仍不能陈述：

- 产品已经解决目标用户需求。
- 用户表达能力、工作绩效或长期记忆得到提升。
- 合成角色实验等同于用户研究。
- 用户留存、推荐或付费意愿已经验证。
