# STG 用户价值验证方案

更新日期：2026-07-20

## 1. 要解决的用户需求

目标用户是知道自己想表达什么、但在求职面试中容易铺垫过长、结构松散或缺少事实支撑的中文求职者。

核心用户任务（JTBD）：

> 当我准备一场重要面试时，我希望用自己的真实回答进行短时、可重复的结构化练习，明确知道哪里影响理解、怎样修改，并确认修改后的表达是否更容易被听懂。

STG 的核心价值不是“生成一份漂亮答案”，而是帮助用户完成：原始回答 → 定位问题 → 理解证据 → 自主修订 → 再次练习。

## 2. 当前产品价值结论

| 能力 | 当前结论 | 证据或限制 |
|---|---|---|
| 完成一次训练闭环 | 已通过 | 题目、提交、反馈、三种修订决策、重新评分和再次练习均可完成 |
| 零成本公开体验 | 已通过 | 本地规则分析，不调用数据库或模型服务 |
| 反馈与用户输入相关 | Day 1 已通过 | 识别核心句位置、引用原文并按最终稿重新计算规则得分 |
| 防止虚构经历 | Day 1 已通过 | 优先重排用户原句；缺少结论时只补充明确的表达框架 |
| 七天个性化训练价值 | 尚未验证 | 需要真实免费模型或更多可靠本地规则，并完成真实用户试用 |
| 能力提升 | 不可宣称 | 当前只有系统评分变化，没有外部标准或长期学习数据 |

当前公开演示只证明“结论先行”这一项训练可以产生可理解的即时价值，不能据此宣称完整 AI 教练或七天课程有效。

## 3. 功能验收与价值验收必须分开

### 功能验收

- 第一次点击提交后 1 秒内进入反馈状态；不得发生无提示刷新。
- 反馈必须引用当前回答，不能返回与输入无关的固定内容。
- 采用、保留、自主编辑三条路径均能完成。
- 最终稿与用户决策一致，重新评分基于最终稿。
- 重复点击和刷新不产生重复决策或额外费用。
- 浏览器控制台无错误；公开演示无外部接口请求。

### 用户价值验收

首轮招募 5–8 名目标用户，每人使用自己真实准备中的面试回答完成一次训练。达到以下门槛才进入 20 人 Beta：

- 至少 80% 能在 8 分钟内独立完成第一次训练。
- 至少 80% 能用自己的话复述系统指出的主要表达问题。
- 至少 70% 认为反馈与自己的原回答相关，相关性评分不低于 4/5。
- 至少 70% 完成一次明确修订，并能说明自己为什么采用、保留或编辑。
- 严重事实虚构为 0；与输入无关的反馈为 0。
- 至少 60% 表示愿意在下一场真实面试前再次使用，并能说出具体使用场景。

“喜欢界面”“看起来专业”不计为价值证据。

## 4. 试用任务与访谈问题

用户任务：

1. 选择一个近期真实面试问题，提交未经润色的原始回答。
2. 阅读反馈，并指出哪条证据最有帮助、哪条不准确。
3. 独立选择采用、保留或自主编辑，完成最终稿。
4. 用另一个回答再次练习同一规则。

访谈只问行为和具体场景：

- 最近一次你因为表达不清而没有答好的问题是什么？
- 这份反馈哪一句改变了你的修改方式？为什么？
- 哪条反馈与你的原回答不符？
- 如果下周有面试，你会在什么时间点打开这个产品？
- 你现在会用什么替代方法？STG 比它少了什么或多了什么？

## 5. 继续、修正或停止的决策规则

- 相关性和事实安全达标，但完成率低：优先修正交互和反馈呈现。
- 完成率高，但用户无法说明学到了什么：反馈缺少教学价值，不能扩大课程。
- 用户只复制建议、不理解修改原因：强化自主编辑和微练习，降低“答案生成器”倾向。
- 少于 60% 愿意在真实面试前再次使用：重新检查目标人群、题目场景和核心问题，不用新增功能掩盖需求不成立。
- 达到门槛后，再扩展到分类表达、完整案例和事实证据；每新增一天都重复同样的价值验证。

## 6. Beta 必须留存的证据

- 首次训练完成率、反馈查看率、修订完成率和再次练习率。
- 每条反馈的相关性评分及错误标注。
- 用户原稿、最终稿、修订决策和版本信息（经用户授权并去标识化）。
- 访谈原始记录和基于记录形成的产品决策。
- 真实延迟、成本和失败率。

履历和案例只陈述实际观察到的数据，不把系统分数变化描述为已证明的能力提升。
